# Packet-First Execution Checklist

> 用途：把 packet-first 架构变成日常可执行工作流，避免停留在“知道原则但落不下去”。

---

## 一、章节启动前检查（Chapter Startup Checklist）

在写新章节前，按顺序检查：

### 1. 是否已有上一章摘要
- 有 → 继续
- 没有 → 先补 `summary`

### 2. 是否已有本章初步框架
- 有 → 继续
- 没有 → 先做本章目标 / 阻力 / 代价 / 结束状态

### 3. 是否已预测本章依赖对象
至少列出：
- 人物
- 空间
- 时间
- 事件
- 伏笔

### 4. 是否已生成 chapter packet
- 有 → 进入写作
- 没有 → 先生成 packet

### 5. 是否已列出本章预计改动哪些卡片
至少确认：
- 哪个人物会变
- 哪个事件会推进
- 哪个空间 / 场景会变
- 哪个时间锚点会更新
- 哪条伏笔会推进 / 延后

---

## 二、默认读取顺序

写新章节时，默认读取：

1. `chapter packet`
2. 上一章结构化摘要
3. 当前活动人物索引
4. 当前活动事件索引
5. 当前活动空间 / 场景索引
6. 当前待回收伏笔索引
7. 再按需读取对应对象卡

默认不读：
- 上一章全文
- 更早章节全文

---

## 三、允许回读前文全文的条件

只有以下情况允许回读：

1. 需要精确复用已发布对白语气
2. 需要核验细微动作连续性
3. 需要确认某段原文措辞
4. 摘要 + packet + 卡片仍不足以支持创作

如果只是“心里不踏实”，不构成回读全文的理由。

---

## 四、正文生成时的执行原则

### 1. 正文优先服从 packet
不是优先服从记忆里的模糊印象。

### 2. 正文优先服从对象卡
如果人物卡 / 空间卡 / 场景卡写得更明确，以卡片为准。

### 3. 当前章只处理当前章目标
不要顺手把后面几章的功能提前塞进来。

### 4. 写的时候同步盯住预计变化项
正文不是自由漫游，而是要完成：
- 人物变化
- 事件推进
- 时间推进
- 空间 / 场景变化
- 伏笔推进

---

## 五、章节写完后的最小回写清单

写完后至少回写：

1. 本章 summary
2. 本章 packet（如有细化）
3. 人物变化记录
4. 事件变化记录
5. 空间 / 场景变化记录
6. 时间锚点
7. 伏笔状态
8. `state.json`
9. `chapter-meta.json`
10. `indexes/`

---

## 六、一句话执行规则

> **没有 summary 不开下一章；没有 packet 不直接写；没有回写就不算真正完成一章。**
